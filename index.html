What's happening on the phone

The phone isn't one program talking SIP the whole time — it's three internal clients, each with its own job and its own server on the other end. Each part below is one of those conversations: request out, response back. In part 3, the SIP client's "server" is actually the SBC — the PBX sits behind it doing the real call-control work — and the dots are paced to the actual four-message SIP handshake. Select any part here, in the layer tabs, or in the step tracker above — everything stays in sync.

Quick translation: the SBC is the front door — it's the only thing the phone ever actually talks to. The PBX is the back office behind that door — it owns the dial plan, voicemail, and call features. If a phone can't register, the problem is almost always between the phone and the SBC, not the PBX.
Common gotcha: a lot of "registration keeps failing" tickets aren't a SIP problem at all — they're NAT. If REGISTER leaves the phone but no response ever comes back, or registration succeeds but drops every few minutes, check the router/firewall in front of the phone before touching anything on the SBC side. Resolving NAT issues

Use this when the flow diagram's Router/NAT flag matches what you're seeing — REGISTER going out with no response, or registration dropping every few minutes. The fix is to move the phone off the default SIP port.

  1. Go to Extensions > System and find the UAD Auto Provisioning Template field.
  2. Add the line below, then Save.
  3. Pick an available port — don't double-assign the same port to two extensions on the same device.
  4. Log in to the Yealink Management Cloud Service.
  5. Remotely reboot the phone so it picks up the new port: Device Management > search for the MAC address > Diagnose > Reboot.

Reading the provisioning log

This is what a healthy phone boot looks like from the provisioning server's side. Half these rows look like errors — most of them aren't. Knowing which red rows are normal is the difference between closing a ticket and chasing a ghost.

Where to find it:

  1. Log in to the PBX.
  2. Go to Reports.
  3. Go to Auto Provisioning.
  4. Look for the MAC address of the phone you're troubleshooting.

The Yealink request sequence, per boot:

  1. Phone asks for MAC.boot — an optional pointer file listing which configs to fetch. Server answers 401 (prove who you are), phone retries with credentials, gets 404 — no boot file exists for this device. That's normal.
  2. Phone falls back to MAC.cfg — the per-device config file itself. Same dance: 401 challenge, retry with credentials, then 200 OK. Config delivered.
  3. The /prov/_directory/ requests are the phone fetching its directory/contacts — separate from config.
What you seeWhat it means
.cfg → 200The row that matters. Config delivered — provisioning worked.
401 then retryNormal digest auth challenge. Only worry if there's no 200 after it.
.boot → 404Normal — no boot file for this device; phone falls back to MAC.cfg.
.cfg → 404Real problem. No config exists for that MAC — check the device record.
401 repeating, no 200Real problem. Credentials rejected — device auth is wrong.
Sound familiar? Challenge → retry with credentials → success is the exact same pattern as the SIP REGISTER handshake on the Flow tab. HTTP and SIP even reuse the same digest auth mechanism — learn it once, read it in two places.

What sip.listen_port actually does

It's the local port the phone's SIP client listens on — and sends REGISTER from. Out of the box, every SIP device on the planet uses 5060. That's exactly why it causes trouble, and why moving off it works.

Yealink phones only. sip.listen_port is Yealink config syntax — it will not work on other phone types. Other vendors have their own parameter for the same setting, so putting this line in a non-Yealink device's template does nothing.
It dodges SIP ALG Many routers ship with SIP ALG on — a "helper" that inspects and rewrites SIP traffic on port 5060, and usually breaks it. Traffic on a non-standard port sails past ALG untouched.
It ends port collisions Several phones behind one router, all on 5060, fight over the same NAT mapping — registrations overwrite each other and drop. A unique port per phone gives each one its own clean path.
Quick rules: one port per extension per site — never double-assign. Avoid 5060 and 5061. The phone must reboot to pick up the change (steps 4–5 above). And document which port went to which extension, or the next tech inherits a guessing game.
Tying it back to the layers: this one setting is the whole lesson in miniature — it's delivered by the provisioning layer (that's why it lives in the UAD template) but it changes registration behavior. Config file carries it in; SIP uses it.
